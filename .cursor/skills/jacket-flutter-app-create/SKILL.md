# Flutter iOS 马甲包创建（集成 app_common）

## 适用场景

- 用户要“生成一个新的 Flutter iOS 马甲包/白包”
- 用户提到“在 `apps/` 目录创建新应用，并集成 `app_common`”
- 用户要按照 `docs/skill需求.md` 的流程自动化生成包

## 目标产物

- 在 `apps/<app_name>/` 创建一个**真实可用**的 Flutter app（iOS 目标可编译）
- **语言要求（硬性）**：生成的 app 需要 **全英文**（UI 文案、页面标题、按钮/提示、空状态、设置项、README 的产品说明与功能清单）。代码技术标识（类名/变量名/文件名/依赖名）保持原样，不要中文化。
- **命名要求（硬性）**：生成的应用名称要尽可能“真实产品化”：
  - 目录名（`apps/<app_name>`）避免包含：`jacket`、`demo`、`test`、`sample`、`example`、`tmp`
  - 产品名（README/页面标题/应用内标题）使用英文 Title Case（1–3 词），避免 snake_case
- 默认情况下（用户未指定主题），由 LLM 自动生成一个合理的 app 主题，并给出完整 UI/配色/交互
- 具备最小闭环能力：可新增/编辑/完成/删除（按主题定义）、本地持久化、基础设置页
- app 集成 `app_common`：启动进入 `BootPage` → `BootCoordinator` 走 remote-config 决策；当决策为 local 时进入“真实应用”主页
- 准备后续步骤：remote-config 随机字段 keyset、README 输出、Info.plist 权限自检

## 输入（与用户对齐）

向用户收集（缺省可由 LLM 自动合理生成）：
- **app_name**：kebab-case 或 snake_case（例如 `food_tool_001`）
- **马甲包类型**：工具包 / 游戏包（缺省则随机选一个并写入 README）
- **bundle id / display name**：若用户没给，可先用占位并提示后续替换
- **应用主题（可选）**：用户可指定（如“喝水记录”）；未指定则由 LLM 自行决定并在 README 说明

## 工作流程（按顺序执行）

### 1) 创建 app 目录与 Flutter 工程

- 确认目标路径不存在：`apps/<app_name>/`
- 若用户未提供 `app_name`，则由 LLM 自动生成一个更像真实产品的目录名（2–3 个英文单词组成的 lower_snake_case），并同时生成一个 Title Case 产品名写入 README
- 创建 Flutter 工程（推荐）：`flutter create apps/<app_name>`
- 保持仓库结构一致：不要把工程创建到仓库根目录

### 2) 集成 `app_common`

- 修改 `apps/<app_name>/pubspec.yaml`：
  - 添加依赖：
    - `app_common: path: ../../packages/app_common`
    - 按需添加 `dio`、`connectivity_plus`、`webview_flutter`、`url_launcher` 等（参考现有 `apps/food_app/pubspec.yaml`）
- 修改入口：
  - `lib/main.dart` 运行 app
  - app 的 home 指向 `BootPage`（可直接参照 `apps/food_app/lib/boot/boot_page.dart` 的结构）

### 3) 拷贝/复用 boot 页面结构（建议最小复用）

- 在新 app 中建立 `lib/boot/boot_page.dart`
  - 使用 `BootCoordinator(remoteConfigClient: RemoteConfigClient(...))`
  - `localHomeBuilder` 指向“真实应用主页”（见第 4 步生成的 home）
- 在新 app 中建立 `lib/boot/remote_config_keys.dart`
  - 先使用 demo 明文字段（便于开发期调试）；马甲包生成时会由专门 skill 替换为随机 keyset

### 4) 生成“真实应用”本体（LLM 决定主题与实现）

#### 4.1 主题选择（未指定时）

- 若用户未指定主题，LLM **必须随机生成一个主题**（不要从固定候选列表里挑）。
- **多样性约束（硬性）**：随机主题必须属于一个明确的“产品类型”，并且与“记录/日志/打卡/追踪”类明显不同（避免总是 tracker/journal）。
  - 例：可以是工具型（decision helper / planner / timer / checklist generator / pack list builder）、学习型（flashcards / spaced repetition）、创作型（prompt studio / storyboard cards）、轻游戏化（daily puzzle / mini challenges）等。
  - 禁止：仅把“记录对象”换个名字但 Home 仍是“列表 + 新建”的同构形态。
- **核心实体差异（硬性）**：核心实体/数据模型必须与上一次生成的 app 明显不同（字段集合与交互语义都要不同）。
- **Home 信息架构差异（硬性）**：Home 必须采用不同的信息架构（例如：仪表盘卡片 + 快捷操作 + 小组件网格；或时间轴/日历视图；或分栏/分段控制；或步骤式向导入口），不能总是“线性列表 + FAB”。
- 随机主题需同时生成以下内容，并写入 README：
  - **产品名**（中英文皆可）
  - **一句话价值主张**
  - **核心实体/数据模型**（例如 Item/Entry/Session 等）
  - **最小闭环功能**：新增/编辑/完成或状态变更/删除 + 列表或统计展示
  - **至少 3 个页面**的信息架构（Home / Detail-Edit / Settings）
- 随机主题约束（必须满足）：
  - 不依赖后端；纯本地可用
  - UI 能体现“像真的产品”（配色系统、空状态、表单校验、交互反馈）
  - 功能闭环清晰、可测试

#### 4.2 UI/信息架构（必须落地）

- 至少包含 3 个页面：
  - Home（核心操作与列表/统计）
  - Detail/Edit（新增/编辑）
  - Settings（必须与主题强绑定的设置项）
- 设计要求：
  - 有明确的色彩系统（primary/secondary/surface/error）
  - 使用 Material 3（ThemeData/useMaterial3）
  - 列表、空状态、表单校验、Snackbar/Dialogs 等交互完备

#### 4.2.1 Settings 页差异化约束（硬性，避免“每个 app 都长一样”）

- Settings 页必须包含**至少 2 个与本 app 产品概念强绑定**的设置项，并且真正影响行为/UI（例如默认模式、排序规则、提醒策略、难度/节奏、单位/阈值、模板偏好、显示密度等），且需持久化。
- Settings 页**禁止**使用同构模板（例如固定的“Appearance 颜色 ChoiceChip + Reset all data + About”三段式）作为主要结构；允许包含“Reset/关于”，但不能成为唯一内容且布局需变化。
- Settings 页建议采用**不同的组件组合/结构**来体现差异：
  - `SwitchListTile` / `RadioListTile` / `SegmentedButton`
  - `Slider` + 实时预览卡片（preview card）
  - “Defaults / Behaviors / Privacy / Advanced”分组卡片（cards）
  - 导入/导出（JSON/文本）或“快速配置”向导入口

#### 4.3 数据模型与持久化（必须落地）

- 使用轻量持久化（推荐 `shared_preferences` 或 JSON 文件）：
  - 列表数据可恢复
  - 设置项可恢复
- 不引入复杂依赖（如数据库）除非主题强需求

#### 4.4 代码结构（建议，但需一致）

- `lib/app/`：主题与路由
- `lib/features/<feature>/`：页面、模型、存储
- `lib/boot/`：启动与 remote-config

### 4.5 生成贴合主题的 App Icon（自动）

- 由 LLM 根据主题产出一段英文 icon brief（1–3 句），包含：
  - subject（图标表达的核心概念）
  - style（flat/vector/gradient/outlined）、mood、主色系
  - 约束：无文字、居中、对比强、缩小后仍可识别
- 自动生成 **1024×1024 PNG**，保存到：
  - `apps/<app_name>/assets/app_icon.png`
- 使用 `flutter_launcher_icons` 写入 iOS/Android：
  - 在 `apps/<app_name>/pubspec.yaml` 增加 `dev_dependencies: flutter_launcher_icons`
  - 配置 `flutter_icons` 指向 `assets/app_icon.png`
  - 运行：`flutter pub get && dart run flutter_launcher_icons`

### 5) 生成 README 基础信息

- 在 `apps/<app_name>/README.md` 写入：
  - 马甲包类型（工具包/游戏包）
  - 应用主题与主要功能说明（LLM 自动生成）
  - remote-config 使用说明（字段映射与响应示例由另一个 skill 负责生成/追加）

### 6) iOS 权限（Info.plist）检查留待专门 skill

- 此 skill 不负责最终补齐所有权限文案；但应确保后续可被 `jacket-ios-privacy-plist-check` skill 扫描与补齐

## 验证

- `cd apps/<app_name> && flutter test`（至少确保工程能解析依赖并编译测试）
- 如需更强验证：`flutter build ios --no-codesign`（环境允许时）

## 注意事项

- 不要把 remote-config 的 JSON key 写死在 `app_common` 内部；应通过 `RemoteConfigKeys` 注入（见本仓库实现）
- 创建马甲包时默认不要启用“兼容旧字段”模式，除非用户明确要迁移窗口
